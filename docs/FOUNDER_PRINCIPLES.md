## Purpose

The Founder Constitution defines the permanent principles that guide Market Pulse AI. These principles should be respected by every future developer, AI assistant, contributor, and release process.

## Constitution

### 1. Production is the Source of Truth

A feature is not complete only because it works locally. Production behavior must be verified for production-facing changes.

### 2. Zero Mock Policy

Production must never display fabricated market data, charts, scores, news, financial values, or recommendations as real.

### 3. Explainability First

Every major output should explain why it exists and what evidence supports it.

### 4. AI Assists. Humans Decide.

Market Pulse AI supports research and education. The user remains responsible for every investment decision.

### 5. Every Score Must Have Evidence

Scores must be interpretable. A score without evidence is not acceptable for a trust-centered financial product.

### 6. Never Fabricate Data

Unavailable data must be shown as unavailable. Missing evidence must be disclosed.

### 7. Quality Before Quantity

A smaller set of reliable, well-explained features is better than a large set of unstable features.

### 8. One Prompt, One Commit, One Deploy, Founder Review, Approve

Future work should follow a reviewable rhythm:

- One Prompt
- One Commit
- One Deploy when production-facing
- Founder Review
- Approve

### 9. Global-First Architecture

The product should be designed for global markets, multiple asset classes, multilingual users, and provider flexibility.

### 10. Long-Term Maintainability Over Shortcuts

Shortcuts that reduce trust, reliability, or clarity are not acceptable.

### 11. User Trust is the Highest Priority

Trust is more important than speed, visual excitement, or feature count.

### 12. Feature Additions Must Solve Real User Problems

New features should help investors understand markets, reduce confusion, manage risk, or save research time.

## Operating Rhythm

The founder workflow is designed to keep work focused, reviewable, and production-aware. Each prompt should produce one logical change. Each commit should be understandable. Each production-facing change should be verified in production before approval.

## Trust Standard

If a decision could reduce user trust, it should be rejected or redesigned.
